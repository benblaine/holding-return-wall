# Holding-Return Wall

A static web app: a responsive grid of ~100 small charts, one per asset. Each tile shows that
asset's **rolling holding returns** at 1–20 year hold lengths. A line is green where that hold made
money, red where it lost. The wall answers, for every asset and every month side by side: *if I had
bought asset X in month T and sold N years later, what would my money have multiplied by?*

> **Past returns, not forecasts.** See the [methodology page](web/about.html).

- **Decoupled batch + static.** A weekly Python ETL emits one JSON file; the browser fetches it and
  renders with a custom `<canvas>` engine. **Zero runtime data-API calls.**
- **No framework, no chart library.** 100 tiles × up to 8 lines is far too much for React/SVG/Chart.js.
  The vanilla canvas renderer + `IntersectionObserver` lazy drawing is the whole reason it's fast.

## Quick start

### 1. Generate the data (Python 3.11+)

```bash
pip install -r etl/requirements.txt
python etl/build_data.py                 # writes web/public/grid_data.json
python etl/validate.py web/public/grid_data.json
```

### 2. Run the frontend (Node 18+)

```bash
cd web
npm install
npm run dev        # http://localhost:5173
npm run build      # -> web/dist (static)
npm run preview    # serve the build
```

## ETL: data sources & env vars

The fetch layer is a **source adapter** selected by `DATA_SOURCE`. The rolling-return maths and the
output JSON shape are identical for every source.

| Env var          | Values              | Default    | Notes |
|------------------|---------------------|------------|-------|
| `DATA_SOURCE`    | `yfinance` \| `eodhd` | `yfinance` | `yfinance` is the zero-cost dev/v1 path. `eodhd` is the licensed production path. |
| `EODHD_API_KEY`  | your key            | _(unset)_  | Required only when `DATA_SOURCE=eodhd`. One key covers equities/FX/crypto/metals. |
| `GRID_DATA_OUT`  | path                | `web/public/grid_data.json` | Output path. `argv[1]` overrides it. |

- **yfinance** — equities/crypto/FX via Yahoo adjusted close (dividends + splits = total return);
  spot metals via stooq.
- **eodhd** — the EOD endpoint with `adjusted_close`, `period=m`. Symbol map: `NVDA.US`, `BTC-USD.CC`,
  `EURUSD.FOREX`, `XAUUSD.FOREX` (spot gold). To switch: `DATA_SOURCE=eodhd EODHD_API_KEY=… python etl/build_data.py`.
- **Baselines** — the S&P 500 **total-return** index (`^SP500TR`) is always pulled from yfinance
  (free, to 1988, not licensed data) even under `eodhd`. US CPI comes from FRED (keyless), falling
  back to the BLS API where FRED is unreachable.

### Hard rules (correctness, non-negotiable — SPEC §2)

Base assets only, no derivatives. Spot metals (XAU/XAG/XPT/XPD), never futures (GC/SI/…) or ETFs
(GLD/SLV). Index funds are not tiles; the S&P 500 is only a benchmark overlay. Total return via
adjusted close, USD throughout. Never fabricate, pad, or extrapolate history — a failed fetch renders
a labelled placeholder, never a guess.

### Known feed gotchas

- **stooq spot metals** are behind a JS bot wall from many networks. When the live metal fetch misses,
  the ETL falls back to a committed **real LBMA gold snapshot** (`etl/fallback/metals.json`) so the
  gold tile renders; silver/platinum/palladium then show as placeholders until a working feed (EODHD)
  is configured.
- **FRED** can be blocked/slow from some networks; the ETL automatically falls back to the keyless BLS
  CPI API.
- **yfinance** may rate-limit (HTTP 429) under heavy use; `validate.py` only WARNs on individual
  misses and FAILs on mass failure, so a partial run never commits broken data.

## Data contract (ETL ↔ frontend) — FROZEN

```jsonc
{
  "assets": [
    { "id": "nvda", "name": "Nvidia", "cls": "equity", "bucket": "generational winner",
      "inc": 1999.04, "p": [[decYear, price], …],
      "r": { "1": [[decYear, multiple], …], "5": […], "20": […] }, "hasData": true },
    { "id": "xagusd", "name": "Silver (spot)", "cls": "metal", "bucket": "precious metal", "hasData": false }
  ],
  "realIds": ["nvda", "btc", "xauusd", …],     // hasData === true; the default selection
  "baselines": {
    "sp500":     { "r": { "1": […], "5": […], "10": […], "20": […] } },  // S&P TR, same rolling shape
    "inflation": { "cum": [[decYear, cpiIndex], …] }                     // cumulative CPI, first = 1.0
  }
}
```

`decYear = year + (month - 0.5) / 12`. Monthly resolution throughout. `cls ∈ {equity, crypto, metal,
currency}`. Raw file ≈ 3.7 MB; **≈ 0.5 MB over the wire** (Vercel brotli).

## Frontend features

- **Sort** — Universe order / Latest / Best-ever / Median / Worst, at a selectable hold length
  (defaults to the longest active). Placeholders always sort last.
- **Benchmark overlays** — toggle `vs S&P 500` and `vs inflation`: a thin reference line per tile at
  the active horizon, plus `beat S&P (Ny) ✓/✗` and `beat inflation ✓/✗` badges.
- **URL state** — the selection, horizons, sort, and overlays encode into the query string, e.g.
  `?a=btc,xauusd,nvda&h=2,4&sort=best5&ov=sp`. Any view is shareable. (No localStorage.)
- **Mobile** — one column; the control bar collapses into a drawer.
- **States** — loading skeleton, empty-selection message, placeholder tiles for feed-pending assets.

## Deploy (Vercel)

The app is fully static. The repo-root `vercel.json` builds the `web/` app
(`npm --prefix web ci && npm --prefix web run build`, output `web/dist`), so Vercel deploys correctly
from the repository root with no manual Root-Directory setting. Connect the GitHub repo so the weekly
data push redeploys.

The **weekly GitHub Action** (`.github/workflows/refresh.yml`) reinstalls the ETL, rebuilds
`grid_data.json`, validates the anchors, and commits **only if the data changed**. For the EODHD
production feed, add an `EODHD_API_KEY` repo secret and set a repo variable `DATA_SOURCE=eodhd`.

## Acceptance anchors (`etl/validate.py`)

S&P 500 long-run CAGR ~9–11% · gold 20-yr multiple strong · BTC 10-yr enormous · a famous compounder
(Monster/Apple) absurd over 20 yr · CPI rising · ≥ ~80 tiles populated. An anchor off by ~10× means
the pipeline is wrong (usually price-return instead of adjusted, or a ticker rename).
